要在代码中使用 HarmonyOS 的沉浸光感（Immersive Light Effects），你需要根据使用的组件类型选择不同的配置方式。沉浸光感从 HarmonyOS 6.1.0（API 23）开始提供，它通过**沉浸式系统材质**和**空间动效**为组件带来通透、精致的视觉效果。<rsup>1</rsup>

## 一、开启沉浸光感

### 1. 应用级开启（推荐）
在 `module.json5` 中配置，对整个应用生效：
```json5
{
  "module": {
    "name": "entry",
    "type": "entry",
    "metadata": [{
      "name": "ohos.arkui.UIMaterial.state",
      "value": "enable" // 可选：default、enable、disable
    }]
  }
}
```

### 2. 组件级开启
#### 方式一：通过通用属性 `systemMaterial`
```typescript
import { uiMaterial } from '@kit.ArkUI';

@Entry
@Component
struct MyPage {
  build() {
    Column() {
      // 为 Column 组件开启沉浸光感
      Column() {
        Text('沉浸光感')
      }
      .width(328)
      .height(56)
      .borderRadius(28)
      .justifyContent(FlexAlign.Center)
      .systemMaterial(new uiMaterial.ImmersiveMaterial({
        style: uiMaterial.ImmersiveStyle.ULTRA_THIN, // 材质样式
        interactive: true, // 启用交互形变
        lightEffect: { color: undefined } // 光感交互反馈
      }))
    }
  }
}
```

#### 方式二：通过组件独有接口
适用于弹窗类组件：
```typescript
// Toast 示例
import { PromptAction, uiMaterial } from '@kit.ArkUI';
```

## 三、材质样式与级别

### 1. 材质样式（ImmersiveStyle）
| 样式 | 说明 | 适用场景 |
|------|------|----------|
| `ULTRA_THIN` | 超薄样式，透明度高 | 浮动工具栏、搜索框 |
| `THIN` | 薄样式，透明度较高 | 需要较强透明的场景 |
| `REGULAR` | 常规样式 | 通用场景 |
| `THICK` | 厚样式，模糊效果强 | 菜单、卡片 |
| `ULTRA_THICK` | 超厚样式，模糊很强 | 弹窗、半模态面板 |

### 2. 材质级别（MaterialLevel）
| 级别 | 说明 | 性能消耗 |
|------|------|----------|
| `ADAPTIVE` | 系统自适应（推荐） | 自动平衡 |
| `EXQUISITE` | 精致效果 | 高 |
| `GENTLE` | 柔和效果 | 中 |
| `SMOOTH` | 流畅效果 | 低 |

## 四、设备兼容性检查

建议在使用自定义材质前检查设备支持情况：

```typescript
import { hdsMaterial } from '@kit.UIDesignKit';
import { uiMaterial } from '@kit.ArkUI';

@Entry
@Component
struct MyPage {
  @State customMaterialLevel: hdsMaterial.MaterialLevel = hdsMaterial.MaterialLevel.EXQUISITE;
  
  aboutToAppear(): void {
    // 查询设备支持的材质类型
    let materialTypes: Array<hdsMaterial.MaterialType> = hdsMaterial.getSystemMaterialTypes();
    
    // 如果不支持 IMMERSIVE，降级到 SMOOTH
    if (materialTypes.indexOf(hdsMaterial.MaterialType.IMMERSIVE) < 0) {
      this.customMaterialLevel = hdsMaterial.MaterialLevel.SMOOTH;
    }
  }
  
  build() {
    Column() {
      // 使用检查后的材质级别
      Button('沉浸光感按钮')
        .systemMaterial(new uiMaterial.ImmersiveMaterial({
          style: uiMaterial.ImmersiveStyle.ULTRA_THIN,
          materialLevel: this.customMaterialLevel
        }))
    }
  }
}
```

## 六、注意事项

1. **版本要求**：需要 HarmonyOS 6.1.0（API 23）或更高版本
2. **性能优化**：低端设备建议使用 `ADAPTIVE` 或 `SMOOTH` 级别
3. **属性冲突**：设置 `systemMaterial` 后，避免同时设置背景色、阴影等样式
4. **深浅色适配**：沉浸光感会自动适配系统深浅色模式
5. **自动反色**：薄材质（`ULTRA_THIN`、`THIN`）可开启 `colorInvert: true` 确保文字可读性

## 七、最佳实践建议

1. **优先使用自适应模式**：`materialType: hdsMaterial.MaterialType.ADAPTIVE`
2. **焦点组件使用**：只在关键交互组件上使用，避免全屏滥用
3. **暗色模式效果更佳**：沉浸光感在深色背景上表现更好
4. **HDS组件优先**：对于导航栏和页签，优先使用 HDS 组件的 `systemMaterialEffect`<rsup>2</rsup><rsup>3</rsup>
5. **降级处理**：使用 `getSystemMaterialTypes()` 检查设备支持情况

通过以上配置，你可以轻松为应用添加高级的沉浸光感效果，提升用户体验和界面质感。

**关于 HarmonyOS SDK 24（API 24）使用官方“沉浸光感”组件的说明**

根据官方文档和开发者实践，**HarmonyOS SDK 24（对应 HarmonyOS 6.1.1）完全可以使用官方的“沉浸光感”组件**。该功能自 API 23（HarmonyOS 6.1.0）起正式引入，并在后续版本中持续增强。

### 一、沉浸光感功能概述
沉浸光感（Immersive Light Sensing）是 HarmonyOS Design System (HDS) 提供的一套高级视觉材质系统。它通过模拟真实物理光照模型，为 UI 组件（如导航栏、页签）带来通透的玻璃质感、细腻的光晕反射以及按压时的弹性光效反馈，显著提升界面的层次感和交互沉浸感。<rsup>1</rsup><rsup>2</rsup>

### 二、版本支持与能力范围
沉浸光感的能力随着 SDK 版本迭代而扩展：

| 版本 | 支持范围 | 核心 API/属性 |
| :--- | :--- | :--- |
| **API 23 (HarmonyOS 6.1.0)** | **HDS 组件**：`HdsNavigation`（顶部导航栏）、`HdsTabs`（底部页签） | `systemMaterialEffect` |
| **API 26 (HarmonyOS 7.0)** | **ArkUI 通用组件**：`Button`、`Column`、`Row`、`Toggle`、`Dialog`、`Popup` 等 | `systemMaterial` |

**结论**：您的目标 SDK 24 位于这两个关键版本之间，因此**支持 HDS 组件（`HdsNavigation` 和 `HdsTabs`）的沉浸光感效果**。若需为更多普通 ArkUI 组件添加此效果，则需要将 `compileSdkVersion` 和 `targetSdkVersion` 升级至 API 26 或更高。

### 三、在 SDK 24 项目中使用沉浸光感（HDS 组件）

#### 1. 环境与依赖确认
确保您的项目满足以下条件：
- **DevEco Studio**：版本支持 HarmonyOS SDK 6.1.0 (API 23) 或更高。
- **项目配置**：在 `build-profile.json5` 中，`compileSdkVersion` 和 `targetSdkVersion` 至少为 `6` (对应 API 23)。
- **模块依赖**：`@kit.UIDesignKit` 已自动包含在 SDK 中，无需在 `oh-package.json5` 中额外声明。

#### 2. 基础使用示例：为 HdsTabs 添加沉浸光感
以下代码展示了如何为底部悬浮页签启用沉浸光感：<rsup>3</rsup>

```typescript
      // 您的 TabContent 内容...
      TabContent() {
        Text('首页')
      }
      .tabBar('首页')

      TabContent() {
        Text('我的')
      }
      .tabBar('我的')

    .barFloatingStyle({
      barBottomMargin: 28, // 悬浮页签距离底部的间距
      // 核心配置：启用沉浸光感
      systemMaterialEffect: {
        materialType: hdsMaterial.MaterialType.ADAPTIVE, // 材质类型：自适应
        materialLevel: hdsMaterial.MaterialLevel.ADAPTIVE // 材质等级：自适应
      }
    })
  }
}
```

#### 3. 关键参数说明
- **`materialType`**：材质类型。
  - `hdsMaterial.MaterialType.ADAPTIVE`：**推荐**。系统根据设备性能自动选择最佳效果。
  - `hdsMaterial.MaterialType.IMMERSIVE`：强制使用沉浸材质（需设备支持）。
- **`materialLevel`**：材质渲染等级。
  - `hdsMaterial.MaterialLevel.ADAPTIVE`：**推荐**。系统自适应。
  - `hdsMaterial.MaterialLevel.EXQUISITE`：精致（效果最强，性能要求高）。
  - `hdsMaterial.MaterialLevel.GENTLE`：柔和（均衡效果与性能）。
  - `hdsMaterial.MaterialLevel.SMOOTH`：流畅（最轻量，兼容低性能设备）。

#### 4. 高级用法：查询设备能力与自定义
如果希望在某些高性能设备上强制使用顶级效果（`EXQUISITE`），应先查询设备支持的能力。

```typescript
import { hdsMaterial } from '@kit.UIDesignKit';
import { BusinessError } from '@kit.BasicServicesKit';

@State customMaterialLevel: hdsMaterial.MaterialLevel = hdsMaterial.MaterialLevel.EXQUISITE;

aboutToAppear(): void {
  // 1. 查询当前设备支持的系统材质类型
  let materialTypes: Array<hdsMaterial.MaterialType> = hdsMaterial.getSystemMaterialTypes();
  // 2. 判断是否支持 IMMERSIVE 类型
  const isImmersiveSupported = materialTypes.includes(hdsMaterial.MaterialType.IMMERSIVE);
  // 3. 如果不支持，则降级到 SMOOTH
  if (!isImmersiveSupported) {
    this.customMaterialLevel = hdsMaterial.MaterialLevel.SMOOTH;
  }
}

// 在 barFloatingStyle 中使用自定义等级
.barFloatingStyle({
  systemMaterialEffect: {
    materialType: hdsMaterial.MaterialType.IMMERSIVE,
    materialLevel: this.customMaterialLevel // 使用动态判断后的等级
  }
})
```

### 四、重要注意事项与最佳实践
1.  **设备兼容性**：沉浸光感效果（尤其是 `IMMERSIVE` 类型）需要设备硬件支持。务必使用 `hdsMaterial.getSystemMaterialTypes()` 查询并做好降级处理。
2.  **性能考量**：`EXQUISITE` 等级效果最华丽，但GPU开销最大。在低端设备或复杂页面上，建议使用 `ADAPTIVE` 或 `GENTLE`。
3.  **系统设置影响**：用户可在系统设置中调整“沉浸光感”强度（强/均衡/弱），这会对应用设置的 `materialLevel` 产生**向下压制**。例如，即使用户设备支持且代码设置为 `EXQUISITE`，若系统设置为“均衡”，实际效果将是 `GENTLE`。
4.  **内容避让**：使用悬浮页签 (`barOverlap(true)`) 时，页面底部内容需要预留空间（如设置 `paddingBottom`），或使用 `expandSafeArea()` 等方法，防止内容被遮挡。
5.  **真机验证**：沉浸光感效果在模拟器上可能无法完整呈现，**务必在真机上进行测试**。

### 五、总结
- **SDK 24（API 24）支持沉浸光感**，您可以立即在项目中使用 `HdsNavigation` 和 `HdsTabs` 组件的 `systemMaterialEffect` 属性来启用该效果。<rsup>4</rsup>
- 对于更广泛的 ArkUI 组件（如 Button、Dialog），则需要升级至 **SDK 26（API 26，HarmonyOS 7.0）** 并使用 `systemMaterial` 属性。
- 开发时，优先采用 **`ADAPTIVE`（自适应）** 模式，以兼顾视觉效果与设备性能，并提供优雅的降级方案，确保应用在不同设备上都有良好的体验。

### 一、理解三个关键的 SDK 版本配置

### 二、HarmonyOS 的兼容性原则

1.  **向后兼容（Forward Compatibility）**：基于**老版本** SDK 开发的应用，默认可以在**新版本**的 HarmonyOS 设备上正常运行。这是系统保证的。<rsup>2</rsup><rsup>1</rsup>
2.  **向前兼容（Backward Compatibility）**：基于**新版本** SDK 开发的应用，若想运行在**老版本**设备上，**开发者必须对新增的 API 进行运行时版本判断和降级处理**。

**结论：** 您用 SDK 24 编译的应用，可以在 `compatibleSdkVersion` 指定的低版本设备上安装运行。但对于 SDK 24 才引入的新组件/API，必须在代码中判断设备版本，并提供降级方案。<rsup>7</rsup><rsup>2</rsup>

**推荐方法（API 26.0.0+ 首选）：使用 `apiAvailable`**


```typescript
// capabilities.ts - 系统能力管理工具类
export class CapabilityManager {
  private static capabilities = new Map<string, boolean>();
  
  // 预检查常用能力
  static async initialize() {
    const commonCapabilities = [
      'SystemCapability.Communication.Bluetooth.Core',
      'SystemCapability.Multimedia.Camera.Core',
      'SystemCapability.Location.Location.Core',
      'SystemCapability.Notification.Notification',
      'SystemCapability.FileManagement.File.FileIO'
    ];
    
    for (const cap of commonCapabilities) {
      this.capabilities.set(cap, canIUse(cap));
    }
  }
  
  static isSupported(capability: string): boolean {
    return this.capabilities.get(capability) || false;
  }
  
  static require(capability: string, featureName: string): void {
    if (!this.isSupported(capability)) {
      throw new Error(`设备不支持${featureName}功能，需要系统能力: ${capability}`);
    }
  }
}

// 使用示例

  'SystemCapability.Multimedia.Camera.Core',
```

沉浸光感（Immersive Light）是 HarmonyOS ArkUI 提供的一套**高品质视觉与动效体系**，它通过**沉浸式系统材质（ImmersiveMaterial）** 与**空间动效**的结合，为应用组件带来通透、精致的视觉表现。它**不是一个可以直接拖拽使用的独立 UI 组件**，而是一套需要通过配置和 API 调用来启用的**系统级材质效果**。<rsup>1</rsup><rsup>2</rsup>

## 🎨 沉浸光感的核心构成

| 能力 | 说明 |
| :--- | :--- |
| **沉浸式系统材质** | 影响组件的背景色、边框、阴影和材质滤镜，呈现层次感和通透感。 |
| **空间动效** | 为组件的弹出过程增添形变、流光等动态表现，使动画更加灵动。 |
| **自适应** | 根据设备算力档位和用户在系统设置中配置的沉浸光感强弱，自动调整表现程度。 |

## 🚀 如何使用沉浸光感？

使用沉浸光感需要完成以下两个步骤：

### 1. 应用级开启（必须）
在项目的 `module.json5`（entry 类型 module）中配置 metadata，全局控制沉浸式系统材质的开启状态。

```json5
// module.json5
{
  "module": {
    "name": "entry",
    "type": "entry",
    // ...
    "metadata": [{
      "name": "ohos.arkui.UIMaterial.state",
      "value": "enable" // 可选值：default、enable、disable
    }],
    // ...
  }
}
```

- **enable**：强制开启沉浸式系统材质。
- **disable**：全局禁用。
- **default**：部分组件（如 Dialog、Toast）在未设置背景色等属性时默认开启。

> **注意**：此配置仅在 `entry` 类型的 module 中生效，且要求 `targetAPIVersion` ≥ 26.0.0。

### 2. 组件级使用
开启应用级开关后，即可在具体的组件上应用沉浸光感效果。根据组件类型，设置方式分为两种：

#### 方式一：通过通用属性 `systemMaterial`（适用于大多数 ArkUI 组件）
```typescript
import { uiMaterial } from '@kit.ArkUI';

@Entry
@Component
struct ImmersiveDemo {
  build() {
    Column() {
      // 对 Column 组件应用超薄材质样式
      Column() {
        Text('沉浸光感示例')
      }
      .width(200)
      .height(80)
      .borderRadius(20)
      .justifyContent(FlexAlign.Center)
      .systemMaterial(new uiMaterial.ImmersiveMaterial({
        style: uiMaterial.ImmersiveStyle.ULTRA_THIN, // 材质样式
        interactive: true,      // 启用按压弹性形变
        lightEffect: { color: undefined } // 启用光感交互反馈（自动颜色）
      }))
    }
    .width('100%')
    .height('100%')
    .justifyContent(FlexAlign.Center)
  }
}
```

**材质样式（ImmersiveStyle）** 从薄到厚共有五档：
- `ULTRA_THIN`：超薄，高度透明（适合浮动工具栏）
- `THIN`：薄，较强透明（适合搜索框、Toast）
- `REGULAR`：常规厚度（通用场景）
- `THICK`：厚，模糊效果强（适合菜单）
- `ULTRA_THICK`：超厚，完全模糊（适合弹窗、半模态页面）

#### 方式二：通过组件独有接口（适用于弹窗类组件）
```typescript
// Toast 示例
import { PromptAction, uiMaterial } from '@kit.ArkUI';

PromptAction.showToast({
  message: '操作成功',
  duration: 2000,
  systemMaterial: new uiMaterial.ImmersiveMaterial({
    style: uiMaterial.ImmersiveStyle.THIN
  })
});

// Popup 示例
import { uiMaterial } from '@kit.ArkUI';

@Entry
@Component
struct PopupDemo {
  @State isShow: boolean = false;

  build() {
    Button('弹出')
      .onClick(() => { this.isShow = !this.isShow; })
      .bindPopup(this.isShow, {
        message: '这是一个带材质的弹窗',
        systemMaterial: new uiMaterial.ImmersiveMaterial({
          style: uiMaterial.ImmersiveStyle.THICK
        })
      })
  }
}
```

#### 方式三：HDS 组件的 `systemMaterialEffect`（适用于 HdsTabs、HdsNavigation）
```typescript
import { hdsMaterial, HdsTabs, HdsTabsController } from '@kit.UIDesignKit';

@Entry

  build() {

      // TabContent...
    }
    .barOverlap(true)
    .barPosition(BarPosition.End)

      }
    })
  }
```

## ⚠️ 注意事项与常见问题

1.  **背景色冲突**：设置 `systemMaterial` 后，建议将组件的 `backgroundColor` 设为 `Color.Transparent` 或移除背景色设置，否则会覆盖材质效果。
2.  **materialColor 透明度**：若使用 `materialColor` 为材质赋色，**必须使用带透明度的颜色**（如 `'#80FFFFFF'`），纯不透明色会完全遮挡材质滤镜效果。<rsup>2</rsup><rsup>1</rsup>
3.  **自动反色（colorInvert）**：仅当材质足够薄（`THIN` 或 `ULTRA_THIN`）且系统沉浸光感配置较强时，组件内的文字颜色会自动适配背景反色，确保可读性。
4.  **设备自适应**：效果会根据设备算力（高/中/低档）自动分级渲染，开发者无需手动适配不同性能设备。
5.  **关闭单个组件的材质**：在 `ENABLE` 模式下，若想单独关闭某个默认开启材质的组件（如 Select），可设置 `systemMaterial(uiMaterial.Material.empty)`。

## 一、声明式UI与状态管理规范

### 1. 严格遵循声明式UI原则
- **UI是状态的函数**：所有界面变化必须通过状态驱动，禁止直接操作DOM或View实例（如`findViewById`、`ref`操作子组件内部）。
- **正确示例**：通过`@State`变量控制`Text`内容。
- **错误示例**：在按钮点击后“直接修改某个Text的文本”。
- 若需获取组件尺寸或位置，应使用`GeometryTransition`或`onAreaChange`回调，而非命令式查询。

### 2. 合理管理状态作用域
- **局部状态**：使用`@State`，仅限当前组件。
- **父子通信**：使用`@Prop`（单向）、`@Link`（双向）。
- **跨组件/跨页面共享**：使用`AppStorage` + `@StorageLink`。
- **复杂对象响应式更新**：配合`@Observed`类与`@ObjectLink`。
- **避免**将所有状态提升到全局，防止状态污染和不必要的重绘。

### 3. 避免在`build`中引入逻辑和副作用
- `build`方法应仅包含UI描述代码，不得包含：
  - 网络请求、文件读写
  - 状态赋值（如`this.count = 10`）
  - 路由跳转（`router.pushUrl`）
  - 定时器、事件订阅
- 这些操作应移至`aboutToAppear`、`onPageShow`或事件回调中，并在`aboutToDisappear`中清理资源。

## 二、ArkTS语法与编码规范

### 1. 命名规范
- **类名**：大驼峰，如`BankAccountManager`。
- **方法名**：小驼峰，动词或动词短语，如`calculateInterest()`。
- **变量名**：小驼峰，如`accountBalance`。
- **常量**：全大写，下划线分隔，如`MAX_TRANSACTION_AMOUNT`。
- **枚举**：全大写，如`TRANSACTION_TYPE`。
- **禁止**使用拼音、无意义命名、系统关键字（如`onClick`）。

### 2. 代码风格与格式
- **缩进**：使用空格（建议2格），禁止使用Tab。
- **分号**：除`build()`外，每行代码必须以分号结束。
- **代码块**：即使只有一行，也必须使用大括号`{}`。
- **条件语句**：避免在控制性条件表达式中执行赋值操作；及时返回，减少嵌套。
- **类型标注**：显式标注函数参数、返回值及重要变量的类型，增强可读性。
- **装饰器顺序**：注意装饰器的应用顺序，不同装饰器可能有不同的执行顺序和影响。

### 3. ArkTS语法约束（违反将无法编译）
- **不支持`any`和`unknown`类型**：必须显式指定类型。
- **不支持解构赋值**：请改用临时变量或逐字段操作。
- **不支持动态字段访问**：禁止`obj["field"]`，请使用`obj.field`。
- **不支持`var`关键字**：请使用`let`或`const`。
- **不支持函数表达式**：请改用箭头函数。
- **不支持`is`运算符**：必须替换为`instanceof`。
- **不支持`with`语句**。
- **不支持`eval`、`new Function`等动态执行代码**。
- **所有`import`语句必须在程序开头**。
- **更多约束**：详见官方ArkTS语法约束列表。

### 4. 代码数量与重复限制
- **函数体不超过80行**：过长函数应拆分。
- **大括号嵌套不超过4层**：通过提前返回等方式减少嵌套。
- **重复代码提取**：三行以上重复三次，或五行以上重复两次，必须提取为共用函数或组件。

## 三、性能优化规范

### 1. 列表渲染
- **少量静态数据**：使用`ForEach`。
- **长列表（>50项）**：必须使用`LazyForEach` + `IDataSource`，否则会导致内存暴涨、滚动卡顿。

### 2. 避免不必要的对象创建
- 避免在高频回调（如滑动、动画）中创建新对象。
- 使用`cachedCount`优化`Tabs`性能。
- 图片加载使用`Image`组件内置能力，避免`base64`。

### 3. 异步与并发
- 大数据处理移至后台任务（`Worker`或`TaskPool`）。
- 使用`DevEco Profiler`定期分析启动时间、帧率、内存泄漏。

### 4. 动画与渲染
- 优先使用HarmonyOS提供的原生动画API和高级模板。
- 优先使用声明式UI和`@State`驱动动画。
- 对于包含复杂子组件的动画，将其设置为`renderGroup(true)`，减少渲染批次。
- **禁止**在动画过程中频繁改变组件的`width`、`height`、`padding`、`margin`等布局属性。

## 四、组件与生命周期规范

### 1. 生命周期正确使用
- `aboutToAppear`：组件即将显示，用于初始化。

### 2. 弹窗与模态层使用规范

- 代码逻辑如果复制三次以上，需要提取共有函数或组件进行复用。

## 一、UI渲染优化

### 1. **布局扁平化**
- **问题**：过深的嵌套层级会增加布局计算时间。
- **优化**：使用扁平化布局，减少不必要的容器嵌套。

```typescript
// ❌ 冗余嵌套（性能差）
Column() {
  Row() {
    Column() {
      Text('Hello')
    }
  }
}

// ✅ 扁平化（性能优）
Text('Hello')
```

- 优先使用 `Column`、`Row`、`Grid`、`RelativeContainer` 等高效布局组件
- 布局层级建议控制在 **5-8层以内**

### 2. **列表性能优化**
- **必须使用 `LazyForEach`** 替代 `ForEach` 处理长列表
- 设置合理的 `cachedCount` 预缓存项数

```typescript
      ItemComponent({ item: item })

.cachedCount(5) // 预缓存屏幕外5个项
```

鸿蒙 ArkUI：V1 与 V2 装饰器全面对比与迁移指南
浏览22发布于2026-06-15 14:58北京
鸿蒙 ArkUI：V1 与 V2 装饰器全面对比与迁移指南
前言
在鸿蒙 ArkTS 开发中，官方在 API 11 / HarmonyOS NEXT 阶段推出了全新的 V2 状态管理体系，对原有 V1 体系进行了系统性重构。两套体系在装饰器命名、响应式粒度、数据流方向等核心设计上均有显著变化。

本文从实际项目出发，系统梳理 V1 和 V2 的差异、各装饰器的使用方法，以及迁移时的注意事项。

一、为什么要有 V2？
V1 体系存在以下痛点：

响应式粒度粗：@State 标记整个对象，对象内任意属性变化都会触发整个组件重渲染，性能差
数据流不透明：@Link 双向绑定容易形成"数据迷宫"，难以追踪变更来源
嵌套对象不响应：@Observed + @ObjectLink 才能处理嵌套对象，使用繁琐
跨组件通信受限：@Provide/@Consume 与 @StorageLink 的能力边界模糊
V2 的目标是：精准响应、单向数据流、更清晰的组件契约。

二、组件声明：@Component vs @ComponentV2
V1 写法
@Component
struct MyCard {
  @State count: number = 0

  build() {
    Text(`${this.count}`)
      .onClick(() => this.count++)
  }
}
V2 写法
@ComponentV2
struct MyCard {
  @Local count: number = 0

  build() {
    Text(`${this.count}`)
      .onClick(() => this.count++)
  }
}
关键区别：

@ComponentV2 是 V2 的入口，只能使用 V2 装饰器（@Local、@Param 等）
@Component 只能使用 V1 装饰器（@State、@Prop 等）
两套体系不可混用，同一个 struct 内不能混合 V1/V2 装饰器
三、组件内部状态：@State vs @Local
V1 — @State
@Component
struct Counter {
  @State count: number = 0  // 整个变量被监听

  build() {
    Button(`count: ${this.count}`)
      .onClick(() => this.count++)
  }
}
V2 — @Local
@ComponentV2
struct Counter {
  @Local count: number = 0  // 组件内部私有状态

  build() {
    Button(`count: ${this.count}`)
      .onClick(() => this.count++)
  }
}
差异对比：

特性	@State（V1）	@Local（V2）
是否可从外部传入	✅ 可以（但不推荐）	❌ 严格私有，不接受外部传入
响应粒度	对象整体	对象整体（配合 @Trace 实现字段级）
语义	组件状态	组件内部私有状态（更明确）
四、父子传参：@Prop / @Link vs @Param / @Event
这是 V1 到 V2 变化最大的地方。

V1 — @Prop（单向）和 @Link（双向）
// 父组件
@Component
struct Parent {
  @State value: string = 'hello'

  build() {
    Column() {
      // @Prop 单向传入，子组件修改不影响父
      ChildA({ value: this.value })
      // @Link 双向绑定，子组件修改同步到父
      ChildB({ value: $this.value })
    }
  }
}

@Component
struct ChildA {
  @Prop value: string = ''  // 拷贝，修改不影响父
  build() { Text(this.value) }
}

@Component
struct ChildB {
  @Link value: string  // 引用，修改直接影响父
  build() {
    Text(this.value)
      .onClick(() => this.value = 'world')
  }
}
V2 — @Param（只读传入）和 @Event（事件回调）
V2 彻底移除了 @Link，统一为单向数据流 + 事件回调：

// 父组件
@ComponentV2
struct Parent {
  @Local value: string = 'hello'

  build() {
    Child({
      value: this.value,
      onValueChange: (newVal: string) => {
        this.value = newVal  // 父组件自己决定是否更新
      }
    })
  }
}

// 子组件
@ComponentV2
struct Child {
  @Param value: string = ''          // 只读，不可直接修改
  @Event onValueChange: (v: string) => void = () => {}  // 事件回调

  build() {
    Text(this.value)
      .onClick(() => this.onValueChange('world'))  // 通过事件通知父组件
  }
}
差异对比：

特性	V1	V2
父→子传值	@Prop（拷贝）	@Param（引用，只读）
子→父通信	@Link（双向绑定）	@Event（单向事件）
数据流方向	双向（@Link）	强制单向
子组件能否直接修改父状态	✅ 能（@Link）	❌ 不能，必须通过 @Event
@Param 使用要点：

标注了 @Require 则调用方必须传入，否则编译报错
子组件不得直接对 @Param 赋值，只能读取
如需在子组件内修改，拷贝到 @Local 变量再操作
五、可观察对象：@Observed vs @ObservedV2 + @Trace
这是 V2 响应式最核心的改进：字段级精准响应。

V1 — @Observed（对象级）
@Observed
class UserInfo {
  name: string = ''
  age: number = 0
  // name 或 age 任一变化，引用此对象的组件全部重渲染
}

@Component
struct UserCard {
  @ObjectLink user: UserInfo

  build() {
    Column() {
      Text(this.user.name)
      Text(`${this.user.age}`)
    }
  }
}
V1 的问题：修改 user.age 会导致显示 name 的 Text 也重渲染，即使 name 没变。

V2 — @ObservedV2 + @Trace（字段级）
@ObservedV2
class UserInfo {
  @Trace name: string = ''   // 只有这个字段变化才触发相关组件更新
  @Trace age: number = 0     // 独立追踪，互不影响
  role: string = 'user'      // 未标 @Trace，变化不触发更新
}

@ComponentV2
struct UserCard {
  @Param user: UserInfo = new UserInfo()

  build() {
    Column() {
      Text(this.user.name)      // 只在 name 变化时更新
      Text(`${this.user.age}`)  // 只在 age 变化时更新
    }
  }
}
差异对比：

特性	@Observed（V1）	@ObservedV2 + @Trace（V2）
响应粒度	整个对象	单个字段（只有 @Trace 字段被追踪）
使用复杂度	需配合 @ObjectLink	直接 @Param 传入即可
嵌套对象	需每层都加 @Observed	嵌套类也加 @ObservedV2 + @Trace
性能	粗粒度，容易过度渲染	细粒度，精准更新
六、跨组件状态：@Provide/@Consume vs @Provider/@Consumer
V1 — @Provide / @Consume
@Component
struct GrandParent {
  @Provide('theme') theme: string = 'light'

  build() { Parent() }
}

@Component
struct Child {
  @Consume('theme') theme: string  // 跨层级直接消费，可直接修改

  build() {
    Text(this.theme)
      .onClick(() => this.theme = 'dark')  // 双向，影响所有使用者
  }
}
V2 — @Provider / @Consumer
@ComponentV2
struct GrandParent {
  @Provider() selectedIndex: number = 0  // 默认用变量名作 key

  build() { Parent() }
}

@ComponentV2
struct Child {
  @Consumer() selectedIndex: number = 0  // 读写，变化同步到 Provider

  build() {
    Text(`${this.selectedIndex}`)
      .onClick(() => this.selectedIndex = 1)
  }
}
差异对比：

特性	@Provide/@Consume（V1）	@Provider/@Consumer（V2）
命名方式	字符串 key	默认变量名（可指定别名）
修改方向	双向（Consume 可直接改）	双向（Consumer 可直接改）
初始值	Consume 不设初始值	Consumer 必须设默认值（Provider 未找到时使用）
七、监听变化：@Watch vs @Monitor
V1 — @Watch
@Component
struct Page {
  @State @Watch('onCountChange') count: number = 0

  onCountChange() {
    console.log('count 变了:', this.count)
  }

  build() { /* ... */ }
}
V2 — @Monitor
@ComponentV2
struct Page {
  @Local count: number = 0
  @Local name: string = ''

  // 监听单个字段
  @Monitor('count')
  onCountChange(monitor: IMonitor) {
    const prev = monitor.value()?.before  // 变化前的值
    const curr = monitor.value()?.now     // 变化后的值
    console.log(`count: ${prev} → ${curr}`)
  }

  // 同时监听多个字段
  @Monitor('count', 'name')
  onAnyChange(monitor: IMonitor) {
    console.log('count 或 name 变化了')
  }

  build() { /* ... */ }
}
差异对比：

特性	@Watch（V1）	@Monitor（V2）
绑定方式	标注在被监听字段上	独立方法，参数指定监听目标
获取变化前后值	❌ 不支持	✅ 通过 IMonitor 获取 before/now
监听多个字段	需要多个 @Watch	一个 @Monitor 可指定多个字段名
监听嵌套字段	❌ 不支持	✅ 支持路径（如 'user.name'）
八、计算属性：getter vs @Computed
V1 — 普通 getter
@Component
struct Page {
  @State firstName: string = '张'
  @State lastName: string = '三'

  // 普通 getter，每次访问都重新计算
  get fullName(): string {
    return this.firstName + this.lastName
  }

  build() {
    Text(this.fullName)
  }
}
V2 — @Computed
@ComponentV2
struct Page {
  @Local firstName: string = '张'
  @Local lastName: string = '三'

  // 自动缓存：只有依赖的字段变化时才重新计算
  @Computed
  get fullName(): string {
    return this.firstName + this.lastName
  }

  build() {
    Text(this.fullName)
  }
}
@Computed 的优势： 结果被缓存，依赖未变化时直接返回缓存值，避免重复计算。这在渲染列表或复杂 UI 时性能收益明显。

九、全局存储：AppStorage vs AppStorageV2
V1 — AppStorage
// 存入
AppStorage.setOrCreate('userInfo', new UserInfo())

// 组件中读取（双向绑定）
@Component
struct Page {
  @StorageLink('userInfo') userInfo: UserInfo = new UserInfo()
}
V2 — AppStorageV2
// 存入/获取（connect 模式，不存在则用工厂函数创建）
const userInfo = AppStorageV2.connect(UserInfo, () => new UserInfo())!

// 组件中读取
@ComponentV2
struct Page {
  userInfo: UserInfo = AppStorageV2.connect(UserInfo, () => new UserInfo())!

  build() {
    // userInfo 是 @ObservedV2 类，@Trace 字段自动响应
    Text(this.userInfo.nickname)
  }
}
差异对比：

特性	AppStorage（V1）	AppStorageV2（V2）
Key 方式	字符串	类型本身（类名作 key）或自定义字符串
响应式	@StorageLink 双向绑定	依赖 @ObservedV2 + @Trace 响应
类型安全	弱类型（any）	强类型（泛型）
十、完整对照表
功能	V1 装饰器	V2 装饰器	说明
声明组件	@Component	@ComponentV2	两套不可混用
内部状态	@State	@Local	V2 严格私有
父→子传值	@Prop	@Param	V2 为只读引用
必传参数	—	@Require @Param	V2 可标注必传
子→父通信	@Link	@Event	V2 强制单向
双向绑定语法	$变量名	—	V2 不支持
可观察类	@Observed	@ObservedV2	—
字段追踪	@Track（可选）	@Trace（显式标注）	V2 只追踪 @Trace 字段
嵌套引用	@ObjectLink	@Param	V2 更简洁
跨层共享	@Provide/@Consume	@Provider/@Consumer	—
监听变化	@Watch	@Monitor	V2 支持获取变化前后值
计算属性	普通 getter	@Computed	V2 有缓存
全局存储	AppStorage + @StorageLink	AppStorageV2.connect()	V2 类型更安全
十一、迁移建议
1. 新项目直接用 V2
V2 是官方推荐的现代写法，新项目无需考虑 V1。

2. 老项目渐进迁移
V1 和 V2 组件可以在同一应用中共存（但不能在同一个 struct 内混用），可以按模块逐步替换。

3. 迁移检查清单
□ @Component        →  @ComponentV2
□ @State            →  @Local（内部状态）
□ @Prop             →  @Param（父传子，只读）
□ @Link             →  删除，改为 @Param + @Event 组合
□ @Observed         →  @ObservedV2
□ @Track / 无标注   →  @Trace（显式标注需要追踪的字段）
□ @ObjectLink       →  @Param（直接传 @ObservedV2 对象）
□ @Provide/@Consume →  @Provider/@Consumer
□ @Watch            →  @Monitor
□ 普通 getter       →  @Computed
□ AppStorage        →  AppStorageV2
4. 常见踩坑
踩坑一：@Param 不可赋值

// ❌ 错误：直接修改 @Param
@Param value: string = ''
this.value = 'new'  // 运行时报错

// ✅ 正确：通过 @Event 通知父组件修改
@Event onChange: (v: string) => void = () => {}
this.onChange('new')
踩坑二：@ObservedV2 类中忘记加 @Trace

@ObservedV2
class Config {
  theme: string = 'light'       // ❌ 没有 @Trace，修改不触发 UI 更新
  @Trace language: string = 'zh'  // ✅ 有 @Trace，修改触发更新
}
踩坑三：V1/V2 组件混用

@ComponentV2
struct MyComp {
  @State count: number = 0  // ❌ V2 组件内不能用 @State
  @Local count: number = 0  // ✅ 正确
}
十二、总结
V2 体系的核心设计理念是：明确的数据流方向 + 精准的响应粒度 + 清晰的组件契约。

单向数据流：通过删除 @Link、强制使用 @Event，让数据流向一目了然
精准响应：@ObservedV2 + @Trace 实现字段级响应，大幅减少不必要的重渲染
类型安全：AppStorageV2、@Require @Param 等让组件接口更加明确
如果你正在开发新的鸿蒙应用，建议直接采用 V2 体系；如果是维护老项目，也可以按模块逐步迁移，两套体系可以在应用层面共存。

## 📦 核心API：AVPlayer（`@kit.MediaKit`）

AVPlayer 提供了完整的播放控制、状态监听、资源管理能力，支持本地文件、网络流媒体（HTTP/HTTPS/HLS）等多种资源。<rsup>2</rsup><rsup>3</rsup>

### 1. 基本使用流程

```typescript
import { media } from '@kit.MediaKit';
import { BusinessError } from '@kit.BasicServicesKit';

// 1. 创建播放器实例
let avPlayer: media.AVPlayer | null = null;

async function createPlayer() {
  try {
    avPlayer = await media.createAVPlayer();
    console.info('AVPlayer创建成功');
    setupPlayerListeners(avPlayer);
  } catch (error) {
    console.error(`创建AVPlayer失败: ${(error as BusinessError).message}`);
  }
}

// 2. 设置事件监听（必须在播放前注册）
function setupPlayerListeners(player: media.AVPlayer) {
  // 状态变化监听
  player.on('stateChange', (state: string, reason: media.StateChangeReason) => {
    console.info(`状态变更: ${state}, 原因: ${reason}`);
    switch (state) {
      case 'idle':
        // 闲置状态
        break;
      case 'initialized':
        // 资源初始化完成，可设置surfaceId（视频）并prepare
        player.surfaceId = getSurfaceId(); // 仅视频需要
        player.prepare();
        break;
      case 'prepared':
        // 准备完成，可开始播放
        player.play();
        break;
      case 'playing':
        // 播放中
        break;
      case 'paused':
        // 已暂停
        break;
      case 'completed':
        // 播放完成
        break;
      case 'error':
        // 错误状态
        break;
    }
  });

  // 错误监听
  player.on('error', (err: BusinessError) => {
    console.error(`播放器错误: ${err.code}, ${err.message}`);
  });

  // 进度更新
  player.on('timeUpdate', (currentTime: number) => {
    console.info(`当前播放位置: ${currentTime}ms`);
  });

  // 时长更新
  player.on('durationUpdate', (duration: number) => {
    console.info(`媒体总时长: ${duration}ms`);
  });
}

// 3. 设置播放资源（支持多种方式）
async function setMediaSource() {
  if (!avPlayer) return;
  
  // 方式一：URL（网络或本地文件）
  avPlayer.url = 'https://example.com/video.mp4';
  // 或本地文件：'file:///data/storage/el2/base/haps/entry/files/video.mp4'
  
  // 方式二：文件描述符（HAP包内资源）
  // const context = getContext(this) as common.UIAbilityContext;
  // const fd = await context.resourceManager.getRawFd('video.mp4');
  // avPlayer.fdSrc = { fd: fd.fd, offset: fd.offset, length: fd.length };
  
  // 方式三：数据源回调（流式播放）
  // avPlayer.dataSrc = { fileSize: -1, callback: (buffer, length, pos) => {...} };
}

// 4. 播放控制
function play() {
  avPlayer?.play();
}

function pause() {
  avPlayer?.pause();
}

function stop() {
  avPlayer?.stop();
}

function seekTo(position: number) {
  avPlayer?.seek(position, media.SeekMode.SEEK_CLOSEST);
}

// 5. 释放资源（重要！）
function releasePlayer() {
  if (avPlayer) {
    avPlayer.release();
    avPlayer = null;
  }
}
```

### 2. 完整示例：音频播放器组件

```typescript
import { media } from '@kit.MediaKit';
import { BusinessError } from '@kit.BasicServicesKit';

@Entry
@Component
struct AudioPlayer {
  private avPlayer: media.AVPlayer | null = null;
  @State currentTime: number = 0;
  @State duration: number = 0;
  @State isPlaying: boolean = false;
  @State volume: number = 1.0;

  aboutToAppear() {
    this.initPlayer();
  }

  async initPlayer() {
    try {
      this.avPlayer = await media.createAVPlayer();
      this.setupListeners();
      
      // 设置音频源（示例使用网络音频）
      this.avPlayer.url = 'https://example.com/audio.mp3';
      
      // 配置音频渲染信息（可选）
      this.avPlayer.audioRendererInfo = {
        usage: audio.StreamUsage.STREAM_USAGE_MUSIC,
        rendererFlags: 0
      };
    } catch (error) {
      console.error(`播放器初始化失败: ${(error as BusinessError).message}`);
    }
  }

  setupListeners() {
    if (!this.avPlayer) return;

    this.avPlayer.on('stateChange', (state: string) => {
      switch (state) {
        case 'prepared':
          this.duration = this.avPlayer!.duration;
          break;
        case 'playing':
          this.isPlaying = true;
          break;
        case 'paused':
        case 'completed':
          this.isPlaying = false;
          break;
      }
    });

    this.avPlayer.on('timeUpdate', (time: number) => {
      this.currentTime = time;
    });

    this.avPlayer.on('error', (err: BusinessError) => {
      console.error(`播放错误: ${err.message}`);
      this.isPlaying = false;
    });
  }

  // 播放控制
  togglePlay() {
    if (!this.avPlayer) return;
    
    if (this.isPlaying) {
      this.avPlayer.pause();
    } else {
      if (this.avPlayer.state === 'prepared' || 
          this.avPlayer.state === 'paused' || 
          this.avPlayer.state === 'completed') {
        this.avPlayer.play();
      }
    }
  }

  // 设置音量
  setVolume(value: number) {
    if (this.avPlayer) {
      this.avPlayer.volume = value;
      this.volume = value;
    }
  }

  // 格式化时间显示
  formatTime(ms: number): string {
    const seconds = Math.floor(ms / 1000);
    const min = Math.floor(seconds / 60);
    const sec = seconds % 60;
    return `${min.toString().padStart(2, '0')}:${sec.toString().padStart(2, '0')}`;
  }

  build() {
    Column({ space: 10 }) {
      // 播放控制
      Row({ space: 20 }) {
        Button(this.isPlaying ? '暂停' : '播放')
          .onClick(() => this.togglePlay())
        
        Button('停止')
          .onClick(() => {
            this.avPlayer?.stop();
            this.currentTime = 0;
            this.isPlaying = false;
          })
      }

      // 进度显示
      Text(`${this.formatTime(this.currentTime)} / ${this.formatTime(this.duration)}`)
        .fontSize(14)
      
      Slider({
        value: this.currentTime,
        min: 0,
        max: this.duration || 1,
        step: 1
      })
      .width('80%')
      .onChange((value: number) => {
        // 拖动结束时跳转
        this.avPlayer?.seek(value, media.SeekMode.SEEK_CLOSEST);
      })

      // 音量控制
      Text(`音量: ${Math.round(this.volume * 100)}%`)
      Slider({
        value: this.volume,
        min: 0,
        max: 1,
        step: 0.1
      })
      .width('60%')
      .onChange((value: number) => this.setVolume(value))
    }
    .width('100%')
    .padding(20)
  }

  aboutToDisappear() {
    if (this.avPlayer) {
      this.avPlayer.release();
      this.avPlayer = null;
    }
  }
}
```

### SoundPool（短音频场景）
适用于按钮音效、提示音等短音频（<5秒），延迟极低（<10ms）。<rsup>4</rsup><rsup>5</rsup>

```typescript
import { media } from '@kit.MediaKit';
import { audio } from '@kit.AudioKit';

class SoundPlayer {
  private soundPool: media.SoundPool | null = null;
  private soundId: number = 0;

  async init() {
    const audioRendererInfo: audio.AudioRendererInfo = {
      usage: audio.StreamUsage.STREAM_USAGE_MUSIC,
      rendererFlags: 0
    };
    
    this.soundPool = await media.createSoundPool(5, audioRendererInfo);
    
    // 加载音频资源
    const context = getContext(this) as common.UIAbilityContext;
    const fd = await context.resourceManager.getRawFd('click.wav');
    this.soundId = await this.soundPool.load(fd.fd, fd.offset, fd.length);
  }

  play() {
    if (this.soundPool && this.soundId > 0) {
      this.soundPool.play(this.soundId);
    }
  }
}
```

## 🔧 常见问题

**Q: 播放网络视频无画面？**
A: 检查网络权限，确保URL可访问，视频格式支持（MP4、HLS等）。<rsup>7</rsup>

**Q: 播放本地文件失败？**
A: 确认文件路径正确，使用正确的URL格式：`fd://` 或 `file://`。

**Q: 如何实现后台播放？**
A: 需要配置后台任务权限，并在`aboutToDisappear`中不释放播放器。

**Q: 播放器状态混乱？**
A: 确保遵循状态机顺序，不要在错误状态下调用方法。
